Versiyon Kontrol Sistemi Nedir?

Versiyon kontrol sistemi, bir sistemdeki verileri, dosyaları, evrakları veya projeleri sürüm sistemi ile yönetmeye yönelik bir sistemdir.
Versiyon kontrol sistemi kısaca VCS, bir veriyi ne zaman eklendiğinin tutun bir hatanın hangi sürümde var olduğunu ve hangi sürümde düzeltiğine kadar meta verilerini görebildiğimiz ve aynı zamanda yönetebildiğimiz araçtır. Eğer VCS olmasaydı dosyalar karışır, hataların tarihlerini görülemez ve iyileştirilmeler yapılamazdı.
Genelde kullanım alanları ise Github veya Gitlab gibi Git LFS[Long File Storge(Uzun Dosya Depolama), Git sistemini dosya sistemidir]'e sahip bütün sunucularda kullanılır. VCS'nin en popüler aracı ise Git'dir. Git, bu tarz süreçleri son derece kolay ve anlışır hale getirmiştir. Ayrıca VCS yapmak kalmaz, aynı zamanda Commit gibi "Şu sürümde bu yapıldı" gibi notlar bırakabilir hale getirmiştir.

VCS Olmadan Çalışmak

Eğer VCS olmasaydı, büyük projeler ve büyük ekiplerin yapmış olduğu devasa uygulamalar hiç bir zaman var olmaz veya günümüzde yapması imkansıza yakın olurdu. Çünkü VCS herkese ortak bir veri sunar. Sürüm bilgisi. Eğer bu bilgi verilmesse dosya karmaşaları, önemli dosyaların hiçe sayılıp üzerine yazılması ve hatalar gibi bir dolu tatsız olayların yaşanmasına yol açardı.
Örnek olarak da Linux çekirdeği örneği verilebilir. Linux, aynı zamanda Git'in yaratıcısı olan Linus Torvalds tarafıdan bir eğlence projesi olarak çıkmış olsada günümüzde milyarlaca bilgisayar, sunucu ve mobil cihazlar gibi akıl almaz bir araca dönüştürmüştür. Ama Linus Torvalds bu önemi tek başına proeje kazandırmadı. Bu sayfanın yazılma tarihi itibariyle 18 binden fazla Contributors yani katkıda bulunanlar vardır. Bu insanlar projede değişiklik yaptıkça sürümler güncellenir ve herkese bildirilir. Ne zaman atıldığı ve not olarak kullanılan Commit'de yazılır. İşte bu 18 binden fazla insan Linux çekirdeğini buzlabınızdaki ısı kontrol sistemlerinden telefonuzdaki Android cihazına kadar yer yere yayılmaya başarmışlardır.

VCS Türleri

VCS türlerini tablo halinde bakalım;
İsmi Tanımı Avantajı Dezavantajı
Yerel Versiyon Kontrol Sistemi Tek bir bilgisayarda versiyon kontrol sistemidir. Tek kullanıcılı ve geliştirici destekler. Bireysel projelerde sıklıkla kullanılır. İnternet veya sunucu ihtiyacı yoktur. Ekipli projeler için elverişsizdir.
Merkezi Versiyon Kontrol Sistemi Proje bir ana sunucuda depolanır ve geliştiriciler o sunucuya bağlanarak dosyaları geliştirir ve kayıt eder. Ekipli projelerde ortak dosya paylaşımı sağlar. Ana sunucu çökerse tüm projenin geliştirme sürecide durur.
Dağıtık Sistemli Versiyon Kontrol Sistemi Git'in kullandığı sistemdir. Geliştiriciler ana projeyi sunucudan çeker, kendi makinelerindeki kopya üzerinde işlemler gerçekleştirirler. Ardından işlemler bittikten sonra geri sunucuya gönderirler. Bu sistem iki aşamalıdır. İlk önce Commit yani not bırakılır, ardından Push ile sunucuya iletilir. Ana sunucu çökse bile geliştiricilerde birer kopaysı olduğu için geliştirme süreci baltalanmaz.s Farkında olmadan bir dosya üzerinde birden fazla geliştirici aynı anda geliştirme yaparsa veri kaybı veya bozulması gözlemlenebilir.

Git Nedir?

Git, VCS için en çok tercih edilen ve kullan bir programdır. Çoğunlukla büyük projelerde veya ekipli projelerde sıklıkla kullanılır. Hatta bu dökümantasyonun Repo ise Github'da bulunmaktadır.
Git, yedi Nisan 2005'te Linus Torvalds tarafından Linux çekirdeği projesi için versiyon yönetim sistemi arayışla başlamıştır. O zamanlar BitKeeper ile çalışmış fakat tersine mühendislikle beraber kendi versiyon yönetim sistemi Git'i geliştirmiştir.

Diğer VCS Programları

Tek VCS programı Git değildir. Diğerlerini tablo şeklinde karşılaştıralım.
İsim Tanım Git ile Karşılaştırma
Apache Subversion Subversion, 2000 yılında CollabNet tarafından versiyon kontrol sistemi programı olarak ortaya çıkmıştır. Git'in aksine dağıtık sistemli versiyon kontrol sistemi yerine merkezi versiyon kontrol sistemi kullanmaktadır.
Mercurial Mercurial, on dokuz Nisan 2005'te çıkışı yapılan bir versiyon kontrol sistemidir. Git ile paralel geliştirilmiş ve neredeyse aynı özelliklere sahiptir. Simgesi "Hg" olup kimyadaki "Merkür" elementinden gelmektedir. Her ne kadar Git ile paralel geliştirimiş olsada, Git her konudu Mercurial'dan çok daha gelişmiş ve oturmuş bir sistemdir. Mercurial'da dağıtık sistemli versiyon yönetim sistemli modelini kullanılır.

Linux

sudo apt install git

MacOS

MacOS kurulumu için şu komutu çalıştıralım;
            
brew install git

Windows

İnen uygulamayı çalıştıralım.
Install kısmı bittikten sonra programın kapanmasını bekliyelim. Eğer uzun bir süre kapanmaz veya donarsa görev yöneticisinden Git indiricisini bulup "Görevi Sonlandır" diyelim.

Kontrol

Yüklenip yüklenmediğini anlıyabilmek adında git --version diyelim ve çıktıyı gözlemleyelim.

İlk Yapılandırma

Git kuruldu fakat yapılandırma yapılmazsa bazı kısımlarda bizi engelliyebilir. Bu sebeble ilk kurulduğunda yapmamız gerek yapılandırma bilgilerini Git'e girelim.

Kullanıcı Adı

Kullanıcı adı, bir repoya bir Push atıcağımız zaman gözükecek olan isimdir. Tabikide Git'de anonim kalamayız çünkü isimsiz biri gelip sizin reponuzu silebilir! Bu sebeble kullanıcı adı yapılandırması yapmadan önce Github üzerinden bir hesap oluşturmanız gereklidir. Hesap oluşturduktan sonra bu yapılandırmayı yapınız.
git config --global user.name "<Github'da açtığınız hesabın ismi>"
        
Artık isminiz gözükmektedir.

E-posta

Tabiki kullanıcı adı yetmez, e-posta adreside lazım. Ancak E-posta adresiniz yetkili olduğunuz repolarda değişiklik yapma hakkına sahip olma veya verme yetkisine verilmesini sağlar. Bu sebeble E-posta önemlidir.
git config --global user.email "<Github'da açtığınız hesabın maili>

Editör

Pek kullanmasak da Git içerisinde kullanıcağımız editörü Git'e belirtmemiz gerekir. Bunun için ben Visual Studio Code seçiceğim, fakat siz dilediğiniz IDE'i kullanabilirsiniz.
git config --global core.editor code # Eğer Visual Studio Code değil ise sistem ortam değişkenlerindeki IDE'niniz yolunu tutan değişkenini ismini kullanmanız gerekir

Branch

Branch, aynı repodaki farklı kısımları belirten bir kavramdır. Ana Branch denilen ve tüm ana kodların bulunan kısımın ismi master olarak adlandırılırdı. Fakat zaman içinde bu alışkanlık değişti ve main ismini aldı, Git'i ilk kurduğumuzda bu ayar master olarak ayarlıdır. Bizler main olarak ayarlamalıyız ki Branch değiştirmekle uğraşmayalım. Bu yapılandırmaya "Varsayılan Branch" denir.
git config --global init.defaultBranch main

Yapılandırmayı Listeleme

Yapılan değişikliklere bakmak istersek git config --list şeklinde gözlemleyebiliriz.

Yapılandırma Seviyeleri

Git'de yapılandırma seviyeli tablo şeklinde aşşağıda verilmiştir.
İsmi Açıklaması Yazılışı
Local Repo klasörü içerisine özel olarak tanımlanabilen yapılandırma seviyesidir. Yapılan değişiklikler sadece yapılan Repo klasörünü etkiler. --local
Global Genel tüm repolardaki yapılandırma düzeyidir. Global'daki yapılandırma, repo klasöründeki Local yapılandırma dosyasının kaynağıdır. --global
System Tüm repoları etkliyen bir yapılandırma seviyesidir. Yapılan değişiklikler tüm repoları etkiler. --system

Git'in Çalışma Mantığı

Snapshots tabanlı yani her kaydetmede değişmiş dosyaları kayıt eden bir yapısı vardır. Değiştirilmemiş dosyalar kopyalanmaz, mevcut Snapshots'a yönlendirilir. Bu sayede Git'e hız ve peformans kazandırır.
Başka bir mimari olan Delta ise anlık verileri almak yerine her bir dosyada "Değişiklik var mı?" diye kontrol eder. Delta, alan açısından verimlidir.
Linus Torvalds bu durumda Git için Snapshots mimarisini tercih etmiştir.

Üç Bölge

Git, bir repoda bizlere üç adet kısım sağlar. Çalışma, Sahneleme ve Depo dizinlerini sağlar.
Çalışma dizini, repo dosyaların olduğu kısımdır. Bu kısım Repo klasörüdür.
Sahneleme dizini, bir sonraki değişikliklerin kayıt edileceği kısımdır. Bu kısıma ".git" klasöründeki bir bölümdür.
Depo dizini, Git'in değişiklik geçimişi depoladığı dizine denir. Bu kısım ".git" klasörünün tamamıdır.

Dosya Durumları

Repo klasöründe her bir dosyanın ve altklasörün bir dosya durumu vardır. Bu durumlar dosya ve klasör üzerinde ne gibi işlemler yapıldığına dair bilgiler sağlamaktadır. Tablo şeklinde dosya durumları verilmiştir.
İsmi Açıklaması
Modified Bu dosya üzerinde değişiklik yapıldı fakat henüz Commit atılmadı
Staged Değişiklikler bir sonraki Snapshots'a hazır durumda.
Committed Atılan Commit yerel repoda başarılı bir şekilde Snapshots'landı.
Untracked Git'in kapsamı dışındaki fakat repo klasöründe bulunan dosyaları ve klasörleri temsil eder.
Tracked Git'in kapsamı altında olan tüm dosyaları ve klasörleri temsil eder.

Yardım Alma

Pek tabi bizler birer robot olmadığımız için bilgileri her zaman hatırlayamayabiliriz. Öyle bir zaman gelir ki her gün çalıştırdığınız ve ezberlediğiniz komut olan git commit komutunu unutursunuz ve projeniz bir kaç gün geç kalır. İşte bu gibi durumları önlemek ve unuttuğunuz komutları tekrar hatırlamak için Git bizlere help parametresi sunar. Help, ya Git'i açıklar yada Git <komut> yazarakta yazdığınız komutun ne anlama geldiğini ve kullanımı sunan bir bir sayfayı taracıda açar. Help komutu ile yardım alalım.
git help # Git'i tanıtır

git commit help # Commit komutunu tanıtır
